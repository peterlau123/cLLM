\begin{quote}
Can I use this for header-\/only libraries? \end{quote}


Yes, however you will need to change the library type to an {\ttfamily INTERFACE} library as documented in the \href{CMakeLists.txt}{\texttt{ CMake\+Lists.\+txt}}. See \href{https://github.com/TheLartians/StaticTypeInfo}{\texttt{ here}} for an example header-\/only library based on the template.

\begin{quote}
I don\textquotesingle{}t need a standalone target / documentation. How can I get rid of it? \end{quote}


Simply remove the standalone / documentation directory and according github workflow file.

\begin{quote}
Can I build the standalone and tests at the same time? / How can I tell my IDE about all subprojects? \end{quote}


To keep the template modular, all subprojects derived from the library have been separated into their own CMake modules. This approach makes it trivial for third-\/party projects to re-\/use the projects library code. To allow IDEs to see the full scope of the project, the template includes the {\ttfamily all} directory that will create a single build for all subprojects. Use this as the main directory for best IDE support.

\begin{quote}
I see you are using {\ttfamily GLOB} to add source files in CMake\+Lists.\+txt. Isn\textquotesingle{}t that evil? \end{quote}


Glob is considered bad because any changes to the source file structure \href{https://cmake.org/cmake/help/latest/command/file.html\#filesystem}{\texttt{ might not be automatically caught}} by CMake\textquotesingle{}s builders and you will need to manually invoke CMake on changes. I personally prefer the {\ttfamily GLOB} solution for its simplicity, but feel free to change it to explicitly listing sources.

\begin{quote}
I want create additional targets that depend on my library. Should I modify the main CMake\+Lists to include them? \end{quote}


Avoid including derived projects from the libraries CMake\+Lists (even though it is a common sight in the C++ world), as this effectively inverts the dependency tree and makes the build system hard to reason about. Instead, create a new directory or project with a CMake\+Lists that adds the library as a dependency (e.\+g. like the \href{standalone/CMakeLists.txt}{\texttt{ standalone}} directory). Depending type it might make sense move these components into a separate repositories and reference a specific commit or version of the library. This has the advantage that individual libraries and components can be improved and updated independently.

\begin{quote}
You recommend to add external dependencies using CPM.\+cmake. Will this force users of my library to use CPM.\+cmake as well? \end{quote}


\href{https://github.com/TheLartians/CPM.cmake}{\texttt{ CPM.\+cmake}} should be invisible to library users as it\textquotesingle{}s a self-\/contained CMake Script. If problems do arise, users can always opt-\/out by defining the CMake or env variable \href{https://github.com/cpm-cmake/CPM.cmake\#options}{\texttt{ {\ttfamily CPM\+\_\+\+USE\+\_\+\+LOCAL\+\_\+\+PACKAGES}}}, which will override all calls to {\ttfamily CPMAdd\+Package} with the according {\ttfamily find\+\_\+package} call. This should also enable users to use the project with their favorite external C++ dependency manager, such as vcpkg or Conan.

\begin{quote}
Can I configure and build my project offline? \end{quote}


No internet connection is required for building the project, however when using CPM missing dependencies are downloaded at configure time. To avoid redundant downloads, it\textquotesingle{}s highly recommended to set a CPM.\+cmake cache directory, e.\+g.\+: {\ttfamily export CPM\+\_\+\+SOURCE\+\_\+\+CACHE=\$HOME/.cache/\+CPM}. This will enable shallow clones and allow offline configurations dependencies are already available in the cache.

\begin{quote}
Can I use CPack to create a package installer for my project? \end{quote}


As there are a lot of possible options and configurations, this is not (yet) in the scope of this template. See the \href{https://cmake.org/cmake/help/latest/module/CPack.html}{\texttt{ CPack documentation}} for more information on setting up CPack installers.
